# PLATFORM_PLAYBOOK
## Как принять решение, какой механизм платформы использовать

> Задача оркестратора: не выполнять задачу самому — найти, что уже умеет платформа.
> Этот документ отвечает на вопрос «почему именно этот механизм сейчас», а не «что умеет каждый инструмент».

---

## Дерево решений

Пройти все вопросы. Ответ ДА добавляет механизм в план — не останавливает проверку.

```
Задача получена
│
├─ Нужна информация?
│   ├─ Она уже есть в системе? ──────────────────► Memory / Project Knowledge / DECISIONS/
│   └─ Нужен актуальный факт из внешнего мира? ──► Web
│
├─ Нужен доступ к внешнему сервису? ────────────────► MCP
│   (GitHub, почта, БД, API — то, что Claude не умеет сам)
│
├─ Нужна работа с файлами / кодом / bash? ──────────► Claude Code
│
├─ Это повторяемая процедура? ──────────────────────► Skill
│   (будет вызываться регулярно — не один раз)
│
├─ Нужен параллельный независимый исполнитель? ─────► Sub-agent
│
├─ Нужен план перед сложным действием? ─────────────► Planning
│
└─ Это долгоживущая роль или продукт? ──────────────► Claude Project
    (специализированный агент, не разовый разговор)
│
▼
Что из платформы НЕ решает задачу?
└─ Только это → Engineering Hand-off
```

## Capability Resolver

> MAIN_ASSISTANT определяет когнитивную потребность в THINK — Resolver выполняет операционный lookup: класс потребности → capability.
> Resolver не "думает" — это маппинг. Когнитивная работа происходит в THINK (Need Discovery). Resolver механический.
> MAIN_ASSISTANT не должен знать что существует WebSearch, MCP Calendar и т.д. — только что возникла потребность класса VERIFY или EXTERNAL CONTEXT.

### Маппинг классов потребностей в capabilities

| Класс (Need Discovery) | Capabilities |
|---|---|
| **REMEMBER** | Memory / DECISIONS/ / STATE.md / filesystem |
| **VERIFY** | WebSearch (текущий default) |
| **EXPLORE** | Sub-agents / параллельный поиск |
| **EXTERNAL CONTEXT** | MCP (Calendar, Gmail, и т.д.) |

**Ownership:** MAIN_ENGINEER. При появлении нового capability (MCP, Skill, плагин) — MAIN_ENGINEER обновляет этот маппинг И раздел **Available Platform Capabilities** ниже. Engineering Hand-off закрывается не только сборкой capability, но и обновлением обоих разделов.

**При отсутствии capability для класса:**
```
Need class: [REMEMBER / VERIFY / EXPLORE / EXTERNAL CONTEXT]
Resolution: не найдена
Signal: [одна строка — чего именно не хватает]
```
THINK принимает ограничение, переходит в JUDGE с маркером НЕ ЗНАЕМ. Маркер становится входом для Engineering Hand-off.

## Available Platform Capabilities

> Спецификация capabilities MAIN_ASSISTANT — что входит в комплектацию и почему.
> Ownership: MAIN_ENGINEER. Обновлять при изменениях конфигурации.
> Последняя верификация: 2026-07-17

### Обязательная комплектация — без этого система не работает

| Capability | Класс | Почему обязательна |
|---|---|---|
| Filesystem (read-only) — DECISIONS/, STATE.md, MAIN_ENGINEER/, AI_Projects/ | REMEMBER | Основа роли Chief of Staff: знать контекст до начала разговора |
| Auto-memory (~/.claude/projects/MAIN-ASSISTANT/memory/) | REMEMBER | Накопление знания о Марии между сессиями |
| WebSearch | VERIFY | Рекомендации по платформам и рынку требуют актуальных данных |

### Опциональные — подключаются по задаче

| Capability | Класс | Когда подключать |
|---|---|---|
| Gmail MCP | EXTERNAL CONTEXT | Задача касается переписки Марии |
| Google Calendar MCP | EXTERNAL CONTEXT | Задачи планирования и расписания |
| Google Drive MCP | EXTERNAL CONTEXT | Работа с документами |
| Sub-agents (Agent tool) | EXPLORE | Широкий параллельный поиск, изоляция объёмных результатов |

### Planned Automations — сейчас существуют как протоколы, не Skills

Эти процессы описаны в MAIN_ASSISTANT_CORE как Reference & Protocols. Они станут Anthropic Skills (SKILL.md + tools/ + examples/) только после реализации.

| Процесс | Текущий статус | Когда станет Skill |
|---|---|---|
| Engineering Hand-off | Протокол в CORE.md | После создания SKILL.md + шаблона |
| Agent Specification | Протокол в CORE.md | После создания SKILL.md + шаблона |
| Session Capture | Протокол в CORE.md | После создания SKILL.md + шаблона |

**Не входит в комплектацию MAIN_ASSISTANT:** Tavily, Exa, Context7, Sequential Thinking, Playwright — это окружение MAIN_ENGINEER для AI_Projects/, не для Chief of Staff.

## Карточки механизмов

Одинаковый формат для каждого. Читать не как справку — как контрольный список перед выбором.

---

### Memory

**Признаки выбора:** нужно вспомнить что-то из прошлых сессий; нужно сохранить что-то для будущих сессий.

**Использовать когда:**
- Информация нужна в нескольких сессиях подряд
- Это факт о Марии, подтверждённый принцип работы, контекст активного проекта
- Наблюдение достигло уровня L1+ и имеет явную ценность для будущего

**НЕ использовать когда:**
- Информация относится только к текущей сессии
- Это черновик, гипотеза или временное состояние задачи
- Наблюдение L0 — записать позже, если повторится

**Типичные ошибки:**
- Писать гипотезы как факты (без маркера уровня)
- Использовать неверный тип (feedback вместо observation)
- Не проверять актуальность — читать как вечную истину
- Писать после каждого разговора без разбора ценности

**Комбинации:** Memory + Projects (агент знает историю), Memory + Web (исследование запоминается), Memory + Skills (процедура применяется с контекстом).

---

### Project Knowledge / DECISIONS/

**Признаки выбора:** задача касается темы, по которой у Марии уже есть зафиксированные решения или материалы.

**Использовать когда:**
- Есть файлы с контекстом проекта, методикой, базой объектов
- Мария уже принимала решение по этой теме — искать в DECISIONS/
- Нужен большой объём контента (до 200K токенов активно, ~10x через RAG)

**НЕ использовать когда:**
- Нужно точное word-for-word извлечение из документа — Knowledge использует semantic search, не exact retrieval
- Документ не структурирован — неструктурированный контент снижает качество retrieval

**Типичные ошибки:**
- Считать что "загрузил документ → Claude знает его дословно" — неверно, это semantic search
- Загружать несвязанный контент — разбавляет релевантность

**Комбинации:** Project Knowledge + Memory + System Prompt = специализированный агент с долгой памятью.

---

### Web

**Признаки выбора:** утверждение зависит от текущего состояния внешнего мира.

**Использовать когда:**
- Возможности Claude, GPT, MCP, Skills, API и других платформ
- Версии продуктов и их ограничения
- Рыночные данные, цены, тарифы
- Новости, релизы, изменения

**НЕ использовать когда:**
- Ответ есть в DECISIONS/ или Project Knowledge
- Задача — reasoning, архитектурный анализ, инженерное решение
- Нужно мнение, а не факт

**Типичные ошибки:**
- Отвечать на вопросы о платформе из training knowledge без Web — данные устаревают
- Использовать Web там, где нужен reasoning — Web даёт факты, не выводы

**Комбинации:** Web + Memory (результат research запоминается), Web + Sub-agents (параллельный research из нескольких источников).

---

### MCP серверы

**Признаки выбора:** нужен доступ к внешнему сервису который Claude не умеет вызвать нативно.

**Использовать когда:**
- Нужно отправить письмо (Gmail MCP)
- Нужен доступ к GitHub issues / PR / репозиторию (GitHub MCP)
- Нужны запросы к БД (PostgreSQL / SQLite MCP)
- Нужен внешний API которого нет нативно

**НЕ использовать когда:**
- Claude умеет это сам нативно — не создавать MCP зря
- Сервис нужен один раз — стоимость подключения не оправдана
- MCP подключён но не используется >2 недель — отключить

**Типичные ошибки:**
- Подключать MCP "на всякий случай" — каждый сервер = тысячи токенов overhead в каждом сообщении (4 сервера ≈ 7,000 токенов, 5+ серверов ≈ 50,000+ токенов)
- Не делать аудит подключённых серверов — они накапливаются и незаметно поедают контекст
- Gmail MCP: 2,640 токенов за 7 инструментов — дорого, использовать только при реальной задаче с почтой

**Комбинации:** MCP + Skills (внешний сервис + процедура работы с ним), MCP + Memory (результат операции сохранить).

---

### Skills

**Признаки выбора:** задача — повторяемая процедура, которую нужно выполнять регулярно одним способом.

**Использовать когда:**
- Процедура повторяется регулярно (Engineering Hand-off, Session Capture, Architecture Gate)
- Нужно кодировать конкретный способ делать стандартную задачу
- Хочется вызывать процедуру через `/skill-name`

**НЕ использовать когда:**
- Процедура одноразовая — не будет повторяться
- Правило ещё нестабильно (L0-L1) — оформить в Skill только при устойчивом паттерне
- Уже есть 10+ Skills — сначала ревизия, потом добавление

**Типичные ошибки:**
- Описывать протоколы текстом в CLAUDE.md вместо Skills — каждый раз строить из головы
- Накапливать Skills без ревизии — больше 12 платят контекстный налог
- Создавать Skill для одноразовой задачи

**Комбинации:** Skills + Projects (агент с встроенными процедурами), Skills + MCP (процедура вызывает внешний сервис).

**Сейчас не оформлены как Skills, но должны быть:** Engineering Hand-off (Mode C), Agent Specification (Mode B), Session Capture, Architecture Gate.

---

### Sub-agents

**Признаки выбора:** задача делится на независимые части которые можно выполнять параллельно.

**Использовать когда:**
- Широкий поиск по filesystem или нескольким источникам
- Параллельный research (несколько тем одновременно)
- Изолированные технические подзадачи
- Нужно защитить главный контекст от объёмных результатов

**НЕ использовать когда:**
- Задача требует единого контекста и постоянного уточнения
- Одноразовая простая задача
- Части задачи зависят друг от друга

**Типичные ошибки:**
- Запускать Sub-agent для задачи которая требует живого диалога
- Не разграничить write-операции — параллельные записи конфликтуют (write сериализуется оркестратором)

**Комбинации:** Sub-agents + Web (параллельный research), Sub-agents + Planning (план → параллельное исполнение частей).

---

### Planning (EnterPlanMode)

**Признаки выбора:** сложная многоэтапная задача, где действие без плана рискует.

**Использовать когда:**
- Перед сложной реализацией — исследовать и предложить план без выполнения
- Перед Mode C Hand-off — понять точно что передавать инженеру
- Когда масштаб изменений неизвестен

**НЕ использовать когда:**
- Простая однозначная задача — план только добавляет трение
- Уже есть готовый план (не планировать дважды)

**Типичные ошибки:**
- Пропускать Planning для многоэтапных задач и держать план в голове

---

### Claude Projects (claude.ai) — роль / агент

**Признаки выбора:** нужна долгоживущая специализированная роль, а не разовый разговор.

**Использовать когда:**
- Нужен агент с постоянным System Prompt между сессиями
- Нужна Knowledge Base + Memory на уровне проекта
- Это роль (репетитор, коуч, ассистент по объектам) — не задача

**НЕ использовать когда:**
- Задача заканчивается за один разговор
- Нужен exact retrieval из документов (Knowledge — semantic search)
- Нужны bash/file/Claude Code tools (Projects их не имеет)

**Типичные ошибки:**
- Создавать Project для разовой задачи
- Загружать в Knowledge неструктурированные PDF — retrieval будет плохим
- Ожидать дословного воспроизведения загруженного документа

**Важно:** Voice в Projects = только voice-to-text INPUT (18 языков, push-to-talk). Native voice OUTPUT отсутствует.

---

### Claude Code tools (файлы / bash / code)

**Признаки выбора:** нужна работа с локальными файлами, кодом, shell-командами.

**Использовать когда:**
- Читать / редактировать / создавать файлы
- Запускать bash-команды, тесты, скрипты
- Навигация по codebase
- Всё что требует реальных действий в системе

**НЕ использовать когда:**
- Задача — мышление, анализ, разговор (не нужен инструмент)

---

### Hooks / Cron

**Признаки выбора:** нужна автоматизация без ручного запуска.

**Hooks** — реактивные (срабатывают на события Claude): session end → capture, tool call → log.
**Cron** — проактивные (срабатывают по расписанию): ежемесячная ревизия STATE.md, напоминание о сроках ревизии гипотез.

**Сейчас не используются, но должны быть:**
- Hook: Session Capture — автоматически при завершении сессии
- Cron: ревизия гипотезы "знание→действие" до 2026-09-16

## Каскад проверки перед Engineering Hand-off

Пройти все вопросы. Добавить в план каждый механизм где ответ ДА.

```
1. Memory / Project Knowledge / DECISIONS/
   Есть ли уже зафиксированный ответ или контекст?

2. Web
   Нужен актуальный факт из внешнего мира?

3. Skills
   Есть ли готовая процедура для этой задачи?

4. MCP
   Нужен доступ к внешнему сервису?
   (проверить стоимость токенов перед подключением)

5. Sub-agent
   Задача делится на независимые параллельные части?

6. Planning
   Нужен структурированный план перед действием?

7. Claude Code
   Нужна работа с файлами, кодом, bash?

8. Claude Project
   Это долгоживущая роль — не разовый разговор?

9. Финальный вопрос
   Чего именно не хватает платформе?
   └─ Это и есть содержание Engineering Hand-off.
       Не "нужен инженер" — а "платформа не умеет X, реализовать X".
```

## Platform Decision — выходной артефакт

После каскада сформировать и показать:

```
Platform Decision

Использовать:
✓ [механизм] — [одна строка: зачем]

Не использовать:
✗ [механизм] — [одна строка: почему не нужен]

Недостающая возможность платформы:
[что именно не умеет] / "отсутствует"

Вывод:
Engineering требуется / не требуется
[если требуется — одна строка: что именно]
```

**Пример — AI-репетитор:**
```
Platform Decision

Использовать:
✓ Claude Project — постоянный System Prompt, роль репетитора
✓ Project Knowledge — учебник, методика, curriculum
✓ Memory — профиль ребёнка между сессиями
✓ Web — проверить текущие возможности Projects и Knowledge

Не использовать:
✗ MCP — внешние сервисы не нужны
✗ Sub-agent — один ребёнок, параллельность не нужна
✗ Claude Code — код не нужен
✗ Voice output — отсутствует нативно (только voice-to-text input)

Недостающая возможность:
точное извлечение упражнений из учебника
(Project Knowledge — semantic search, не exact retrieval)

Вывод:
Engineering пока не требуется.
Layer 1 (Project + Knowledge + Memory) достаточен.
Если exact retrieval критичен — Layer 2 (структурированный markdown-шаблон).
```

## Platform Decision Record — обязательная запись

**Когда обязательна:** перед Engineering Hand-off, при запуске нового агента или проекта, при любом архитектурном решении.

Формат — заполнить полностью, не сокращать список "Рассмотрено":

```
Platform Decision Record

Задача:
[одна строка — что решается]

Рассмотрено:
☐/✓ Memory          — [рассмотрен / нужен / не нужен]
☐/✓ Project Knowledge — [рассмотрен / нужен / не нужен]
☐/✓ Web             — [рассмотрен / нужен / не нужен]
☐/✓ Skills          — [рассмотрен / нужен / не нужен]
☐/✓ MCP             — [рассмотрен / нужен / не нужен]
☐/✓ Sub-agent       — [рассмотрен / нужен / не нужен]
☐/✓ Planning        — [рассмотрен / нужен / не нужен]
☐/✓ Claude Code     — [рассмотрен / нужен / не нужен]
☐/✓ Claude Project  — [рассмотрен / нужен / не нужен]

Решение:

Используем:
[механизм 1] + [механизм 2] + ...

Не используем:
[механизм] — [одна строка почему]

Почему:
[2-3 строки: что определило выбор]

Недостающая возможность платформы:
[что не умеет платформа] / "отсутствует"

Engineering Hand-off:
требуется / не требуется
[если требуется — одна строка: что именно реализовать]
```

**Хранение:** Platform Decision Record не сохраняется отдельно — остаётся в сообщении как рабочий артефакт сессии. Если по его результатам принимается долгоживущее архитектурное решение — в соответствующий проект переносится только итоговое Architecture Decision, не весь процесс выбора механизмов.

**Назначение записи:** не отчёт — рабочий артефакт. Мария видит логику решения. Claude в следующей сессии видит почему выбран этот путь и не начинает заново.
