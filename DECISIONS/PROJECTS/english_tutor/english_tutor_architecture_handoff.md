# English Tutor — Architecture Handoff

> Source of Truth проекта. Читать перед любым следующим шагом — не историю переписки.

---

## 1. Цель проекта

Персональная система обучения английскому для сына Марии (идёт в 8 класс, лето 2026).

**Задача:** закрыть хвосты из прошлых классов + подготовить к учебному году. Оценка 5, но есть пробелы.

**Что строим:** Claude Project с набором markdown-файлов, работающий как AI-репетитор с памятью об ученике. Не чат с AI. Не Learning Engine. Не линейное следование учебнику. — Система, которая знает конкретного ребёнка и адаптирует путь под него.

---

## 2. Подтверждённые решения (LOCKED)

### Центр системы — Student Digital Twin, не учебник и не Learning Engine

**Решили:** архитектура строится вокруг модели ученика. Учебник — авторитет по содержанию, не по темпу.

**Почему:** GPT-репетитор без модели ученика придумывал примеры вместо учебника — не потому что плохой, а потому что не было Source of Truth об ученике. Учебник без диагноза = линейное прохождение вне зависимости от пробелов.

**Что отвергнуто:**
- Learning Engine как центр → ошибочная классификация системы; цепочка идёт не туда
- Линейное следование учебнику без диагностики → не закроет хвосты, только добавит материал
- Хранение ошибок как ошибок → нужен диагноз (почему), а не журнал (что)

---

### Два режима, не один

**Решили:** Summer Recovery + School Mode существуют параллельно.

- **Summer Recovery** — диагностика пробелов, индивидуальный маршрут, нелинейный порядок
- **School Mode** — домашние задания строго по учебнику, линейная программа

**Почему:** смешение режимов создаёт конфликт целей. В Recovery ребёнок идёт по слабым местам. В School Mode — строго по расписанию класса.

**Что отвергнуто:** единый режим «учимся по учебнику» → не решает задачу summer recovery.

---

### Политика учебника

**Решили:** учебник — авторитет по **содержанию**, не по **темпу**.

Система может:
- сократить упражнения, если критерии освоения выполнены

Система не может без Марии:
- менять порядок тем
- пропускать темы целиком

**Почему:** педагогический авторитет учебника сохраняется. Автономия системы ограничена темпом, не содержанием.

---

### Критерии освоения темы (все четыре, не один)

1. Independent practice 80%+ в двух разных сессиях
2. Transfer — применил правило в новом контексте
3. Elaborative interrogation — объяснил своими словами корректно
4. Retention check — retrieval через 3–4 дня, 80%+

**Почему четыре:** один критерий (только практика) не гарантирует долгосрочное удержание. Transfer и elaborative interrogation проверяют глубину, а не запоминание формулировки.

---

### Слоистая реализация (Layer decision)

**Решили:** Layer 1 сейчас, Layer 2+ только при наблюдаемой поломке.

- **Layer 1 (сейчас):** Claude Project + markdown-файлы
- **Layer 2 (если Claude ломается):** БД для student_model
- **Layer 3 (если нужна автоматизация):** Agent через MAIN_ENGINEER

Триггер для Layer 2+: наблюдаемая проблема (контекст переполняется, retrieval ненадёжен), не гипотетическая.

## 3. Архитектура Layer 1

### Поток работы

```
Ученик → Student Model → Decision Engine → Lesson → Observation → Update Student Model
```

### Файлы в Claude Project

**`student_model.md`** — когнитивный профиль ученика (не журнал ошибок):
- Паттерны ошибок с диагнозом (почему ошибается, не что)
- Как лучше понимает (аналогии / правила / таблицы / примеры из жизни)
- Тревоги и уверенность по темам
- Динамика по сессиям

**`curriculum.md`** — структурированный учебник:
- Шаблон с пронумерованными темами и упражнениями
- Не PDF — разбитый на единицы, доступный для точного retrieval
- Статус освоения каждой темы

**`lesson_state.md`** — состояние текущего урока:
- Какая тема сейчас
- Какие упражнения пройдены
- Наблюдения этой сессии (до записи в student_model)

**`decision_rules.md`** — правила Decision Engine:
- Как выбрать следующую тему
- Когда тема считается освоенной
- Как обрабатывать повторяющиеся ошибки

### Четыре Source of Truth

| SoT | Авторитет |
|-----|-----------|
| Учебник | Содержание и порядок тем |
| Сам ребёнок (ответы, паузы, попытки) | Сырые данные наблюдений |
| Teacher Book | Педагогические рекомендации |
| Student Model (`student_model.md`) | Накопленный диагноз |

**Конфликты:** при конфликте между учебником и Student Model — учебник сохраняет авторитет по порядку тем; Student Model может обосновать сокращение упражнений.

## 4. Ограничения платформы (Claude Projects, Layer 1)

**Project Knowledge = semantic/vector search.** Не точный retrieval. При запросе «упражнение 5b из раздела 3» Claude найдёт семантически близкое, а не точное совпадение. Вывод: curriculum.md должен быть структурирован с однозначными метками (не просто нумерация страниц), чтобы уменьшить неоднозначность поиска.

**Context window competition.** Все файлы в Project Knowledge конкурируют за контекст. При росте файлов (особенно student_model.md) ранние записи могут вытесняться. Триггер для Layer 2: заметная деградация качества retrieval student_model.

**Нет персистентного состояния между чатами.** Каждая новая сессия начинается с Project Knowledge. lesson_state.md должен явно закрываться в конце каждого урока (наблюдения → student_model), иначе данные теряются.

**MCP — слишком дорого для Layer 1.** 4 MCP-сервера ≈ 7K токенов overhead, 5+ серверов ≈ 50K+. Для репетитора это неоправданная нагрузка на контекст. Оценивать только при переходе в Layer 3.

**Источник:** исследование платформы, проведённое 2026-07-16 в рамках PLATFORM_PLAYBOOK.md.

## 5. Открытые вопросы

### Учебник (критичный)

**Почему открыт:** без учебника нельзя структурировать curriculum.md и написать System Prompt.

**Что нужно проверить:** название, издательство, класс. Желательно — цифровая версия или содержание с разбивкой по разделам.

**Эксперимент:** получить учебник → попробовать структурировать одну главу → проверить, насколько хорошо Claude её retriev'ит.

---

### Мотивация ученика

**Почему открыт:** не знаем, кто принял решение учиться — сын сам или Мария.

**Что нужно проверить:** если решение навязано — нарратив и gamification важнее педагогической точности. Если сын сам хочет — можно строить на академическом прогрессе.

**Что узнать:** во что играет сын (ключ к нарративу). Кто инициировал.

---

### Gamification

**Почему открыт:** не знаем достаточно об ученике, чтобы выбрать нарратив.

**Что нужно проверить:** после прояснения мотивации — нужна ли вообще и какая.

**Не включать в Layer 1** до прояснения этих данных.

## 6. Что НЕ делать

**Layer 2 (БД для student_model) сейчас** — Layer 1 ещё не запускался. Триггер для Layer 2: наблюдаемая поломка retrieval, не гипотетическая нагрузка.

**Agent через MAIN_ENGINEER** — нет технической задачи. Вся логика умещается в System Prompt + markdown.

**MCP-сервер для учебника** — токен-стоимость неоправданна на Layer 1. Учебник входит как структурированный markdown-файл в Project Knowledge.

**Собственная БД** — то же что Layer 2. Преждевременно.

**PDF учебника как Project Knowledge** — Claude не умеет делать точный retrieval из PDF с нумерованными упражнениями. Учебник нужно разбить на текстовые блоки вручную.

**Gamification без данных об ученике** — нарратив без знания интересов ребёнка = мусор.

**Линейное прохождение учебника** — не закрывает summer recovery.
